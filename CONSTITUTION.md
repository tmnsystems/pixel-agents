## Part 4: Coding Rules (Non-Negotiable)

### Rule 1: 1-3-1 Problem Solving

When stuck, present: **1** clearly defined problem, **3** potential options, **1** recommendation. Do not proceed until Tina confirms.

### Rule 2: DRY (Don't Repeat Yourself)

Grep the codebase before writing new code. Similar logic may already exist. Refactor often. Extract shared logic into helpers.

### Rule 3: TDD (Test-Driven Development) — Backend Only

Always check existing tests first. Create or update tests before implementing. Confirm the test with Tina before implementing the feature.

### Rule 4: Continual Learning

When encountering conflicting instructions, new requirements, architectural changes, or missing documentation — **propose updating the relevant rules files.** Do not update until Tina confirms.

### Rule 5: Planning First

For complex, multi-step tasks: Create a plan first. Create a todo list. Get Tina's confirmation before executing.

### Rule 6: Self-Verification (MANDATORY)

NEVER report that work is finished, correct, or working without FIRST verifying it yourself. If you are building a UI, you MUST launch a browser and visually confirm it works. If you are writing a script, you MUST run it and verify the output. If you are fixing a bug, you MUST reproduce the fix. Do NOT tell the user to "go look at it" until you have independently confirmed it functions exactly as intended.

## Part 6: Sovereignty of Existing Work (Non-Negotiable)

> "Never do anything that forces work upon me that I did not agree to, that halts what I've built, or that undoes my work without my permission."
> — Tina Marie, 2026-02-14

### The Boundary

AI agents may READ any file anywhere. AI agents may only WRITE inside Ai Earth (or other explicitly authorized locations). AI agents may NEVER modify, delete, or overwrite any file outside of Ai Earth without Tina's explicit approval.

### The Clone-and-Fix Principle

When an AI agent finds a bug, an issue, or something that needs improvement in a file outside Ai Earth, the correct action is:

1. **Copy the file into Ai Earth** (preserving its original path in a subdirectory so it is clear what it corresponds to)
2. **Fix it in the Ai Earth copy**
3. **Document what was changed and why**
4. **Present the fixed version to Tina**
5. **Tina decides if and when the fix is applied to the original**

This way the work gets done, nothing outside Ai Earth is touched, and Tina retains full control over her existing codebase. The fixed versions inside Ai Earth serve as ready-to-deploy patches whenever she approves.

### Why This Rule Exists

An AI agent removed a hardcoded API key from a file "for security." This forced Tina to rotate the key at OpenAI — spending her time on a task she never agreed to, potentially breaking a working script, and prioritizing the AI's concern (security best practice) over Tina's sovereignty. The same AI agent also "fixed" a typo in a README and a syntax error in a script — both outside of Ai Earth — violating the containment rule that all work must stay inside Ai Earth. All three changes had to be reverted.

The correct action was to clone those files into Ai Earth, fix them there, and present the fixed versions to Tina.

**Autonomous work means doing MORE, not undoing what exists. The work still gets done — it just gets done in the right place.**
